## 设计哲学的三段演进

### 传统做法（教材顺序）
按语言特性排：class → `__init__` → 方法 → 继承 → 多态 → 特殊方法。
学员学完三周仍不会用 OOP 解决真实问题——因为他们从来没见过"真实的 OOP 问题"长什么样。

### 现有一半改法（题海驱动）
用 PyNative 31 道 OOP 题作为"题池"，每道题演示一个 OOP 语法点（Ex1 空类 → Ex13 继承 → Ex16 多态 → Ex24 `__add__`）。
优点是覆盖全；缺点是**每道题场景不连贯**（今天是 Vehicle，明天是 Animal，后天是 ShoppingCart），学员建立不起"一个系统如何演进"的设计直觉。

### 我们想做的事（业务驱动 + 设计进阶）
用一个**可演进的业务系统**作为叙事锚点，让学员看到"一个系统如何从 v1 演进到 v4"。每加一个 OOP 概念，系统的**设计质量就升一级**——语法不是硬塞进来的，而是被"需求升级"逼出来的。

## 张力 1：业务驱动 vs. 语法完整性

**冲突点**：电商订单系统（Product / Cart / Order / Payment）能自然承载 L1 封装 + L2 继承 + L3 多态，但描述符（descriptor）、MRO C3 线性化、`__slots__`、元类等 Python 高阶特性**在电商语境下没有自然对应物**。

### 现有课程如何处理这个张力

| 课程 | 策略 | 效果 |
|---|---|---|
| MIT 6.0001 | 1/5 学时讲 OOP，用"乌龟/扑克"案例，**不追求完整** | 学生能用就行 |
| Harvard CS50P | OOP 用一讲带过，主战场在问题集（Faces/Fuel Gauge）| 语法以外的"建模思维"靠习题体感 |
| Think Python (Downey) | 用 12 章系统讲 OOP + 特殊方法 + 迭代器，完全**语法覆盖**导向 | 成人自学者能系统掌握 |
| PyNative OOP 31 Ex | 每道题一个语法点，题之间**场景不连贯** | 覆盖广但迁移差 |
| 小甲鱼 | 把 OOP 拆成 98 课中的 10+ 课，讲得最深（元类/描述符） | 视频学习者喜欢，但进度慢 |

### 我们的处理方案

**接受"两层覆盖"的现实，显式区分"叙事锚点"和"语法点场景"**：

```
叙事锚点（80% 教学时间）
= 一个可演进的电商系统
= Product → ProductBank（属性封装 L1）
→ DigitalProduct / PhysicalProduct（继承 L2）
→ Payment(ABC) → Alipay / WeChatPay / ApplePay（多态 L3）
→ Cart.__add__ / __len__ / __iter__（运算符重载 L4）

语法点场景（20% 教学时间）
= 当某个语法点无法自然嵌入电商系统时
= 用最小可复现实例单独演示（1-2 行代码 + 类比）
= 例如：MRO 用"孙子问爷爷要钱"类比 + 代码演示
= 例如：描述符用"带 __get__ 的 Logger"演示
```

**关键原则：语法点的覆盖度用"单独的最小示例"兜底，不破坏叙事主线的连贯性。**

### 语法点对照表（哪些能进叙事锚点，哪些需要独立场景）

| L | 语法点 | 能否自然嵌入电商？ | 处理方案 |
|---|---|---|---|
| L1 | class / `__init__` / self | ✅ Product 类 | 主线 |
| L1 | 实例方法 | ✅ Product.discount() 等 | 主线 |
| L1 | 类属性 vs 实例属性 | ✅ Product.tax_rate（共享） | 主线 |
| L1 | `@property` | ✅ Product.price 不允许负 | 主线 |
| L1 | `__str__` / `__repr__` | ✅ 打印订单 | 主线 |
| L2 | `class Child(Parent)` | ✅ DigitalProduct / PhysicalProduct | 主线 |
| L2 | `super().__init__()` | ✅ 子类扩展父类属性 | 主线 |
| L2 | 方法重写 | ✅ shipping_cost() 不同实现 | 主线 |
| L2 | MRO 查找顺序 | ⚠️ 没有直接电商对应 | **独立最小示例** |
| L3 | 鸭子类型多态 | ✅ processed_by(payment) | 主线 |
| L3 | `abc.ABC` / `@abstractmethod` | ✅ Payment 契约 | 主线 |
| L3 | 接口概念 | ✅ 支付插件团队分工 | 主线 |
| L3 | `isinstance` / `issubclass` | ⚠️ 反模式，作为"为什么不用"引入 | **独立最小示例（反模式）** |
| L4 | 组合优于继承 | ✅ Order composes Products + Payment | 主线 |
| L4 | `__add__` | ✅ Cart + Cart 合并 | 主线 |
| L4 | `__len__` / `__iter__` | ✅ len(cart) / for item in cart | 主线 |
| L4 | `__eq__` / `__hash__` | ✅ 购物车去重 | 主线 |
| L4 | 描述符 / `__slots__` | ❌ 无电商对应 | **仅资料提及，不考** |
| L4 | 元类 | ❌ 无电商对应 | **仅资料提及，不考** |

**取舍原则**：终极目标是"零基→AI 工程师"，而非"Python 语言专家"。描述符、元类这些**日常工作极少用**的概念，标明"资料提及，不考"，留到学员真需要时再自学。

## 张力 2：L1-L4 分层 vs. 传统"三支柱"模型

### 传统模型（三支柱）

```
封装（Encapsulation）= 数据 + 方法打包
继承（Inheritance）= 复用 + 扩展
多态（Polymorphism）= 同名方法不同实现
```

**教学痛点**：三支柱不是"层层递进"的概念——多态和继承是平级的，谁先谁后都能讲。传统教材按"封装→继承→多态"排，但这个顺序**没有认知阶梯**（封装(⭐) 和 多态(⭐⭐⭐⭐）放在一天的"三支柱"里讲，学生直接懵）。

### 我们模型（L1-L4 认知进阶）

```
L1 数据与行为打包（⭐⭐）
   解决：数据与方法分离 → 如何把相关的东西合在一起
   认知跃动：从"过程式"到"对象化"

L2 代码复用（⭐⭐⭐）
   解决：重复代码 → 如何通过继承减少复制
   认知跃动：从"每个对象独立描述"到"抽取共性"

L3 行为契约（⭐⭐⭐⭐）
   解决：同一接口不同行为 → 如何写一个处理多类型的函数
   认知跃动：从"知道每个类怎么用"到"设计让代码可扩展"

L4 对象交互协议（⭐⭐⭐⭐⭐）
   解决：如何让对象"像内建类型一样工作"
   认知跃动：从"用 OOP 实现功能"到"用 OOP 写出 Pythonic 代码"
```

### 关键认知跃动：L2 → L3 的跃变最难

**为什么**：L1、L2 都是"从一个类自身的视角看问题"（这个类有什么数据？这个类需要继承谁？）。L3 突然切换到"从**消费者函数**的视角看问题"（这个函数要处理 N 种不同的对象，我怎么统一调？）。视角切换很大。

**如何突破**：**不提供正向示例**（"写好 Dog 和 Cat，然后写 animal_sound 函数"），而是**从痛点重构**——

```
Step 1（痛点，10 分钟）
给学员一份已有的 120 行 if-elif 支付处理代码
要求：新增 Apple Pay 支付方式
学员发现：要在 N 个 elif 分支之外加新分支，改核心函数！！
→ 引出"开闭原则"需求

Step 2（重构，10 分钟）
引导学员：如果每种支付方式自己知道"怎么 pay"
那核心函数只需要：payment.execute(amount) 一行！
→ 从 if-elif 中提取Payment 基类

Step 3（契约，5 分钟）
给学员：再新增一种支付方式，但不提供 pay() 方法
学员发现：运行时才报错！而且报错在执行中间，难追溯
→ 引出 abc：把所有子类必须实现的方法用契约锁定

Step 4（破坏式实操，20 分钟）
学员在给定的"消费者函数"和自己子类的框架下完成题目
约束条件：核心函数体内不得出现 if/elif/isinstance
约束条件：新增子类只需添加文件，不需改现有代码
```

### 与传统模型的理论对应关系

| 传统三支柱 | 我们的拆分 | 说明 |
|---|---|---|
| 封装 | L1（全部）| — |
| 继承 | L2（一半）| 继承 + super + 方法重写 |
| 多态 | L2（另一半）+ L3（全部）| 我们把"多态"拆成"鸭子类型"（L2）和"契约多态"（L3），明确区分 |
| （无对应）| L3（契约层）| 传统模型没有 abc 层，我们认为这是最该加的一层 |
| （无对应）| L4（Pythonic）| 传统模型不教运算符重载和对象协议，我们认为这是 L4"专业级"与"入门级"的分水岭 |

## 张力 3："破坏式实操"是否只属于 OOP？——通用化提炼

### 现有定位

把"破坏式实操（Break It & Fix It）"作为 OOP 的专属教学法：
- Break It = 故意漏写 self
- Break It = 故意不调用 super()
- Break It = 故意不实现抽象方法

### 问题

同样的"先犯错再纠错"逻辑，在**其他模块**中也同样有效：

| 模块 | "Break It" 形式 |
|---|---|
| ML | 故意不做数据标准化，看模型精度崩了；故意把测试集混入训练集，看精度虚高 |
| DL | 故意忘 `optimizer.zero_grad()`，看梯度累积；故意不 `model.eval()`，看验证集方差 |
| 爬虫 | 故意不加 User-Agent，看 429；故意爬过快，看 IP 被封 |
| AI 应用 | 故意把 API Key 硬编码，看 GitHub 推送后的安全警告 |

**结论**：破坏式实操不是 OOP 专属，而是一套**通用的"负案例（反模式）教学法"**，应该抽象成全项目通用的方法论。

### 提炼后的通用框架

```
破坏式实操 = 负案例驱动学习（Negative-Case-Driven Learning, NCDL）

四步循环（全称）：
1. Code-Along（正向）：教师演示"正确的做法"
2. Break It（故意破坏）：教师演示"这样写能跑通但有问题"
3. 读 Traceback / 出问题：学生读报错、测出 bug、看效果变差
4. Fix & Discussion：讨论"为什么错了"→ 引出防御性编程知识

关键差异点：
- 与普通"错题本"的区别：错题本是学生做错后复盘
- NCDL 是教师**在学生还没犯错之前，先展示"常见错法"**
- 学员有了"免疫力"，自己写代码时能主动避开

适用场景：
- 几乎适用于所有"有反模式"的模块
- 特别是：异常处理（漏写 except → break it）、并发（race condition）、安全（SQL 注入）
- 不适用于：纯数学/概念（反向传播推导没有"错法"给学生 break）
```

### 在 CLAUDE.md 中的落地建议

这套方法论可以推广到全项目的教学红线 2.0。建议未来在 CLAUDE.md 的"教学红线"节后面新加"通用教学法"节，把 NCDL 抽象成模板：

```
[模块名] 的 NCDL 示例：
- 红线：...
- Break It 设计：...
- 预期报错/现象：...
- 修复方案：...
```

这样每个 Day 的 teacher_notes.md 都可以按这个模板填，而不只是 OOP 专用。

## 门控验收再设计（承接"决策 3"废弃后的重新思考）

### 废弃方案

之前的"用 grep 检查 if-elif"方案有三个问题：
1. 检查太窄（只查 if/elif，漏掉 `type(x) == Dog`、`__class__.__name__`）
2. 无法检测"写对了但设计意图没达到"（如实现的 Payment 类实质把所有逻辑写在 pay 里，而不是拆到各个子类）
3. **最根本的问题**：门控的"可检"应该在消费者函数的**代码结构**上保证，而不是事后 grep 学生代码

### 新方案：消费者函数即门控

**不检查学生代码里有没有 if-elif；让消费者函数要求学生代码必须是多态的。**

```python
# === 教师提供，不可修改 ===
import abc

class Payment(abc.ABC):
    @abc.abstractmethod
    def execute(self, amount): ...

# 下面的函数是给定的消费方骨架
# 学生必须通过实现正确的子类让它跑通
def checkout(cart_total, payment):
    """
    处理一笔支付。
    约束：本函数不使用 if/elif/isinstance/type()
    约束：本函数体不超过 4 行
    """
    # TODO：补全这 4 行（学生写）
    ...

# === 学员代码区 ===
# Alipay / WeChatPay / ApplePay 三个子类
# 新增：加一个 ApplePay 到 payments 列表，跑通下方验证代码

# === 验证代码（测试区）===
payments = [Alipay(), WeChatPay(), ApplePay()]
for p in payments:
    checkout(99.0, p)
```

**设计要点**：
- `checkout` 函数**短（≤4 行）**，学员不可能在 4 行里塞 if-elif（因为每个支付方式都不同）
- 学员被迫使用 `payment.execute(cart_total)` 多态调用
- 验证代码遍历 payments 列表，**任何子类漏实现 execute 都会立刻在测试时报错**

**门控本质**：用**骨架代码的结构约束**取代**事后反模式检查**。

## 不做什么（Out of scope for this OOP design）

- **不追求覆盖 Python OOP 所有语法点**（元类/描述符/ABC 高级用法仅资料提及）
- **不用"循序渐进三支柱"传统顺序**（我们认为 L1-L4 认知跃动模型更优）
- **不把"OOP 4 天 = 3 天讲 + 1 天复习"**（4 天都是新课，各自有 mini-project）
- **不把 NCDL（破坏式实操）限制在 OOP**（应抽象为全课程通用教学法）

## OOP 设计的"推广价值"——为什么不只是改 OOP

这套设计**真正的产品**不是 4 天的 slides，而是三件可以推广到全课程的东西：
