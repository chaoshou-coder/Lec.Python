# 02 · 开源课程调研方法 —— 如何搜集和分析非商业开源课程

> 用途:为课程设计找到高质量参照,避免"闭门造车"。
> 适用场景:新课程启动、某一模块需要外部素材、竞品分析。

### 1.2 为什么商业化课程不宜直接参照?

商业化课程(Udemy / 极客时间 / 慕课网等)的目标函数不同:

| 维度 | 开源/藤校 | 商业化 |
|---|---|---|
| **目标** | 知识体系完整 | 完课率、好评率、销量 |
| **深度** | 宁可难也要讲透 | 宁可浅也要"听得懂" |
| **习题** | 有挑战,鼓励犯错 | 平滑,避免学员卡住 |
| **更新** | 学术驱动,慢但准 | 市场驱动,快但可能跟风 |
| **认证** | 无或学术证书 | 付费证书(卖点) |

**结论**:商业化课程适合"看市场热点",不适合"抄知识体系"。
我们的做法:用藤校课程定骨架,用商业化课程定热点,用社区(GitHub/知乎)定习题素材。

## 2. 如何提取一门课的"章节顺序"和"知识点粒度"

### 2.1 提取章节顺序

**Step 1 · 找到官方 Syllabus / Schedule**
- MIT OCW:看 "Calendar" 或 "Syllabus" 页
- Coursera:看 "Syllabus" 或 "Weekly Schedule"
- 教材:看目录(Table of Contents)

**Step 2 · 列出每周主题**
把每周的主题写成一行,忽略具体习题和阅读材料。

示例 —— MIT 6.0001 提取结果:
```
W1: What is computation? / Branching and Iteration
W2: String Manipulation, Guess-and-Check, Approximations
W3: Decomposition, Abstraction, Functions
W4: Tuples, Lists, Aliasing, Mutability
W5: Recursion, Dictionaries
W6: Testing, Debugging, Exceptions
W7: Object Oriented Programming
W8: Python Classes and Inheritance
W9: Understanding Program Efficiency
W10: Searching and Sorting
```

**Step 3 · 标注每个主题的"核心知识点"**
把主题翻译成具体的知识点,粒度参考"能出一道题"。

示例:
- "Branching and Iteration" → if/elif/else, while, for, range
- "Decomposition, Abstraction, Functions" → def, return, 参数, 作用域

### 2.2 提取知识点粒度

**判断标准**:一个"知识点"应该满足:
- 能在 15-30 分钟讲完
- 能出至少 2 道练习题
- 有明确的"会/不会"判断标准

**太粗的例子**(需要拆分):
- ❌ "Python 基础" → 拆成:变量/类型/运算符/控制流/函数
- ❌ "数据结构" → 拆成:字符串/列表/字典/集合/元组

**太细的例子**(需要合并):
- ❌ "print 函数" + "input 函数" + "type 函数" → 合并为"输入输出与类型查看"
- ❌ "append" + "insert" + "extend" → 合并为"列表 CRUD"

## 3. 如何对比多门课的顺序差异并提取共识

### 3.1 对比方法

**Step 1 · 选 5-10 门目标课程**
覆盖:藤校 2-3 门 + 经典教材 2-3 门 + 中文热门 2-3 门

**Step 2 · 建立"知识点 × 课程"矩阵**
行 = 知识点,列 = 课程,单元格 = 该课程在第几周讲这个知识点。

**Step 3 · 计算"共识顺序"**
对每个知识点,取所有课程的"周次中位数" → 得到共识顺序。

**Step 4 · 标注"离群点"**
某门课的顺序明显偏离中位数 → 分析原因(可能是创新,可能是失误)。

## 4. 调研数据来源的可靠性评估

### 4.1 可靠性分级

| 级别 | 来源类型 | 示例 | 可信度 |
|---|---|---|---|
| A | 官方课程页面 | MIT OCW, Coursera Syllabus | 95%+ |
| B | 权威教材目录 | Think Python, PCC | 90%+ |
| C | 社区整理 | GitHub Awesome, 知乎专栏 | 70-85% |
| D | 二手转述 | 博客总结,短视频 | 50-70% |
| E | 营销内容 | 课程宣传页,付费推广 | < 50% |

### 4.2 交叉验证原则

- **至少 3 个独立来源**确认同一结论,才能写入课程设计
- **A 级来源**可直接引用;B 级需要交叉验证;C/D 级仅作参考
- **离群点**必须找到原始来源,不能只看转述

### 4.3 常见数据陷阱

| 陷阱 | 案例 | 避免方式 |
|---|---|---|
| 把"课程宣传"当"课程大纲" | 某课程宣传"从入门到精通",实际只教基础 | 找官方 Syllabus |
| 把"学员评价"当"课程质量" | 好评多不代表内容深 | 看大纲,不看评分 |
| 把"更新日期"当"内容新鲜度" | 2024 年更新但内容还是 Python 2 | 看具体内容,不看日期 |
| 把"课程时长"当"内容量" | 100 小时课可能 80 小时是废话 | 看知识点密度 |
