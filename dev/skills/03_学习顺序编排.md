# 03 · 学习顺序编排 —— 如何编排 60 天的学习顺序

> 用途:在知识依赖图的基础上,安排具体的学习顺序,平衡认知负荷和教学效率。
> 适用场景:新课程编排、现有课程压缩/扩展、高负荷日调整。

---

## 1. 三大编排原则

### 1.1 工具先行(Tools-First)

**核心思想**:先教"能做出有用东西"的工具,再教"底层原理"。

**为什么**:零基础学员的最大动力来源是"我能做出东西了"。
如果前 10 天都在讲"变量/类型/运算符",学员看不到 AI 的影子,流失率极高。

**具体策略**:
```
Day 1: print/input → 立刻能写"自我介绍程序"
Day 2: 数据类型 + 运算符 → 立刻能写"简易计算器"
Day 3: 条件分支 → 立刻能写"BMI 计算器"
Day 4: 循环 → 立刻能写"九九乘法表"
Day 5: 字符串 → 立刻能写"手机号脱敏"
Day 6: 列表 → 立刻能写"购物清单"
Day 7: 函数 → 立刻能写"购物车程序"
```

**反例(传统教材顺序)**:
```
Ch1: Python 概述 / 历史 / 安装
Ch2: 变量 / 类型 / 运算符
Ch3: 字符串(详细)
Ch4: 列表(详细)
Ch5: 元组 / 集合 / 字典
Ch6: 条件分支
Ch7: 循环
...
→ 学员学了一周还不知道能做什么
```

**工具先行的代价**:学员对底层原理理解不深。
**解决方案**:螺旋复访(见 1.3),后续模块会反复用到这些工具,自然加深理解。

### 1.2 脚手架递进(Scaffolded Progression)

**核心思想**:每节课只引入一个新抽象层,其余都是已学内容。

**为什么**:认知负荷理论指出,人脑同时处理的新概念 ≤ 2 个。
如果一节课同时引入"列表 + 列表方法 + 列表生成式 + 深浅拷贝",学员会崩溃。

**具体策略**:
```
Day 6: 列表 CRUD (只教创建/索引/append/pop)
Day 7: 列表方法 (只教 sort/reverse/extend)
Day 8: 列表生成式 (只教 [x for x in ...])
Day 9: 深浅拷贝 + 二维列表
```

**反例(传统教材顺序)**:
```
Ch4: 列表(创建/索引/切片/CRUD/方法/生成式/深浅拷贝/二维)
→ 学员一次被轰炸,什么都记不住
```

**脚手架递进的代价**:课程进度变慢。
**解决方案**:每节课配足练习(6 道当堂练),确保"学一个会一个"。

### 1.3 螺旋复访(Spiral Revisit)

**核心思想**:核心概念在三个复杂度层级复访,每次加深理解。

**为什么**:一次学习无法形成长期记忆,需要在不同场景下反复遇到。

**具体策略(以"函数"为例)**:
```
第一层(Day 7-10):函数基础
  def add(a, b): return a + b
  → 理解"输入 → 处理 → 输出"

第二层(Day 26-31):nn.Module
  class Net(nn.Module):
      def __init__(self): ...
      def forward(self, x): ...
  → 理解"类中的函数(方法)"

第三层(Day 37-43):LoRA Config
  lora_config = LoraConfig(
      r=16,
      target_modules=["q_proj", "v_proj"],
      lora_alpha=32,
  )
  → 理解"配置对象 + 函数式调用"
```

**螺旋复访的关键**:
- 每次复访必须**明确提到**与前一层的联系
- 每次复访必须**增加新的复杂度**,不能简单重复
- 每次复访之间**间隔 7-14 天**,利用间隔效应

## 2. 认知负荷评估方法

### 2.1 每天 ≤ 2 新概念

**新概念定义**:学员从未接触过、需要建立新心智模型的知识。

| 类型 | 例子 | 是否新概念 |
|---|---|---|
| 全新抽象 | 函数、类、张量、Attention | ✅ 是 |
| 已学抽象的新用法 | 列表的 sort 方法(已学列表) | ❌ 否 |
| 语法糖 | 列表推导式(本质是 for 循环) | ❌ 否 |
| 工具/库 | requests、BeautifulSoup | ⚠️ 看情况 |

**评估方法**:每天备课时,列出所有"新概念",超过 2 个就要拆分或延后。

### 2.2 高负荷日识别与缓冲

**高负荷日标准**:
- 新概念 ≥ 3 个
- 或包含"抽象度极高"的概念(如梯度下降、Attention、LoRA)

**我们课程的高负荷日**:
| Day | 主题 | 新概念数 | 负荷等级 |
|---|---|---|---|
| Day 32 | 梯度下降 + 反向传播 | 3 | 🔴 极高 |
| Day 44 | Attention 机制(Q/K/V) | 3 | 🔴 极高 |
| Day 51 | LoRA 原理与实现 | 3 | 🔴 极高 |
| Day 26 | 张量 + 自动微分 | 2 | 🟡 高 |
| Day 37 | HuggingFace 入门 | 2 | 🟡 高 |
| Day 53 | RAG 原理 | 2 | 🟡 高 |

**缓冲策略**:
```
高负荷日前一天:只复习,不学新概念
高负荷日当天:放慢节奏,增加示例,减少习题量
高负荷日后一天:综合练习,巩固理解
```

示例 —— Day 32(梯度下降)的缓冲安排:
```
Day 31:复习张量运算 + 自动微分(不学新概念)
Day 32:梯度下降(3 新概念,但节奏放慢,只讲 4 道题)
Day 33:综合练习(用梯度下降训练一个简单模型)
```

### 2.3 "消费 → 加工"模型

**核心思想**:先教"使用已有工具"(消费),再教"理解底层原理"(加工)。

**为什么**:零基础学员无法直接理解底层原理,需要先"看到效果"再"理解原因"。

**具体策略(以字符串为例)**:
```
Day 5:消费字符串
  - 用 replace() 做手机号脱敏
  - 用 split() 做 CSV 解析
  - 用 f-string 做格式化输出
  → 学员"会用",但不知道底层

Day 8:加工字符串
  - 讲字符串的不可变性
  - 讲编码(UTF-8 / ASCII)
  - 讲字符串驻留(interning)
  → 学员"理解底层"
```

**消费 → 加工的时间间隔**:3-5 天最佳。
太短:学员还没"用够",不理解为什么要加工。
太长:学员已经忘记消费阶段的内容。

## 3. 我们课程的具体顺序决策记录

### 3.1 为什么 ML 在 DL 前?

**决策**:Module 1(机器学习,Day 18-25)在 Module 2(深度学习,Day 26-31)前。

**理由**:
1. **知识依赖**:DL 的评估指标(准确率/召回率/F1)在 ML 已经教过
2. **认知负荷**:ML 的算法(线性回归/决策树)比 DL 的算法(反向传播/CNN)直观
3. **工具依赖**:sklearn 的 API 设计(`.fit()`/`.predict()`)是 PyTorch 的铺垫
4. **行业共识**:MIT 6.0002 → CS229 → CS231n 的顺序都是 ML 在 DL 前

**反方意见**:有些课程(如 fast.ai)主张"先 DL 后 ML",理由是"先看到效果"。
**我们的回应**:fast.ai 面向有编程基础的学员,零基础学员需要先建立"什么是模型"的直觉。

### 3.2 为什么 LLM 微调在 Transformer 后?

**决策**:Module 3(NLP 与 Transformer,Day 32-36)在 Module 4(LLM 微调,Day 37-43)前。

**理由**:
1. **知识依赖**:LoRA 微调的对象是 Transformer 架构的模型,不懂 Transformer 就无法理解 LoRA 在微调什么
2. **认知负荷**:Transformer 是 NLP 最复杂的架构,需要先单独讲清楚
3. **工具依赖**:HuggingFace Transformers 库的核心就是 Transformer 模型

**如果反过来会怎样?**
学员会"照抄微调代码",但看不懂报错,无法调试,无法选择模型。

### 3.3 为什么爬虫在 LLM 微调后?

**决策**:Module 5(Web 爬虫,Day 44-50)在 Module 4(LLM 微调,Day 37-43)后。

**理由**:
1. **知识依赖**:爬虫需要 HTTP 协议 + requests 库,这些在 Python 核心已经教过
2. **认知负荷**:爬虫相对独立,不需要 ML/DL 基础
3. **教学节奏**:LLM 微调是"重头戏",爬虫是"轻松模块",放在后面调节节奏
4. **实用价值**:爬虫可以为 LLM 微调收集数据,形成闭环

### 3.4 为什么 AI 应用开发在最后?

**决策**:Module 6(AI 应用开发,Day 51-60)在最后。

**理由**:
1. **知识依赖**:需要所有前置 Module 的基础(ML/DL/NLP/LLM/爬虫)
2. **综合性质**:本 Module 是"把所有知识串起来"的收口
3. **就业导向**:学员最后能拿出"AI 应用作品",对求职最有帮助
4. **动力维持**:最后 10 天是"冲刺阶段",学员看到自己的作品会有成就感

### 3.5 Python 核心从 21 天压缩到 10 天

**决策**:原 21 天 Python 核心压缩为 10 天(Day 1-10)。

**理由**:
1. **目标导向**:学员目标是 AI,不是 Python 专家,不需要"精通 Python"
2. **题量保证**:每节 2-3 道核心题 + 选做题,保证吸收
3. **螺旋复访**:Python 核心知识会在后续 Module 反复用到(如 NumPy 用列表知识)
4. **工具先行**:压缩的关键是"只教必要的,不贪全"

**压缩方法**:
- 当堂练从 6 道减为 2-3 道核心题
- 其余题目标为"选做"
- 删除冗余知识点(如元类/描述符/装饰器深入)
- 把"字符串提升""列表进阶"等移到后续 Module 的"消费 → 加工"阶段

**风险**:压缩后学员基础不牢。
**缓解**:每节课配"课前测查",不过关的学员需要补课。

## 5. 编排的迭代机制

学习顺序不是一次性产物,需要持续迭代:

1. **每次学员卡住**:检查是否认知负荷过高 → 拆分或延后
2. **每次教学进度超前**:检查是否有冗余 → 压缩或删除
3. **每次技术更新**:检查是否有新知识点需要加入 → 增节点
4. **每轮教学后**:根据学员反馈调整顺序

**版本管理**:建议用 Git 管理学习顺序,每次修改写清楚"为什么改"。
