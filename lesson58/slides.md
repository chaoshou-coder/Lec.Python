## 1. 第一讲(20 分钟) —— 四个期末作品方向

### 知识点 1.1 方向总览

期末作品分四个方向,学员根据兴趣和职业目标选一个:

| 方向 | 典型作品 | 亮点技术 | 适合人群 |
|---|---|---|---|
| A. AI 应用 | RAG+Agent 问答 | RAG+多工具 | 大多数同学 |
| B. LLM 微调 | 领域垂直助手 | LoRA+评测 | 想深入 DL |
| C. 爬虫+数据 | 舆情分析系统 | 爬虫+可视化 | 喜欢数据 |
| D. ML 工程 | 端到端 ML 平台 | Pipeline+部署 | 想看工程化 |

> 四个方向工作量相当,验收 checklist 的核心指标一致(功能完整、
>    技术深度、文档清晰)。选哪个取决于个人的"故事线"。

### 知识点 1.2 方向 A —— AI 应用(主推)

**示例作品**:企业内部知识库问答系统

用户场景:员工上传公司 wiki/PDF,用自然语言提问,系统给出引用原文
    的答案,并能调 API(查 CRM、发邮件)。

技术栈:FastAPI + Chroma + LangChain Agent + Streamlit + Docker

预期功能:
- 文档上传 → 切片 → 嵌入 → 入库
- RAG 问答 + 来源引用
- 至少 2 个工具(搜索 + 数据库)
- 流式输出 + Docker 部署

> 这是 7 天课程的"集大成者",选了它等于把 Day 51-57 做一遍完整的。

### 知识点 1.3 方向 B —— LLM 微调

**示例作品**:医疗问诊助手

用 LoRA 微调一个 7B 模型,使其回答医疗问题时符合专业规范。

技术栈:PyTorch + PEFT(LoRA) + SwanLab 评测 + vLLM 部署

难点(收益也大):
- 数据准备(病历脱敏 + 问答对构建)
- 微调训练(显存受限时用 QLoRA)
- 评测对比(Baseline vs 微调后)

> 选这个方向的同学 Day 55-57 用于复习 DL 基础,Day 58-60 全力微调。

### 知识点 1.4 方向 C —— 爬虫 + 数据

**示例作品**:电商舆情监控(品牌口碑仪表盘)

技术栈:Scrapy 爬虫 + LLM 情感分析 + Streamlit 可视化

关键流程:
1. 定时爬取评论/微博
2. 用 LLM(或传统模型)打情感标签
3. 统计趋势 + 关键词云

> 适合对数据更感兴趣、想转数据分析方向的同学。

### 知识点 1.5 方向 D —— ML 工程

**示例作品**:端到端房价预测平台(上传数据 → 训练 → 部署 → 预测)

技术栈:FastAPI + sklearn/pandas + MLflow 实验追踪 + Docker

关键特性:
- 支持多种算法对比
- 实验记录和版本管理
- API 接收特征返回预测值

> 适合已有 ML 基础、想补工程化短板的同学。

## 3. 第二讲(20 分钟) —— 项目结构最佳实践

### 知识点 3.1 标准化项目结构

无论哪个方向,项目结构都遵循这个骨架:

```
my_capstone/
├── README.md            # 项目说明(第一眼见!)
├── requirements.txt     # 依赖
├── .env.example         # 密钥模板(提交!)
├── .gitignore           # 含 .env
├── docker-compose.yml   # 一键启动
├── docs/
│   ├── architecture.md  # 架构图 + 说明
│   └── api.md           # 接口说明
├── src/
│   ├── __init__.py
│   ├── main.py          # FastAPI 入口
│   ├── agents/          # Agent 逻辑
│   ├── rag/             # RAG 管道
│   └── utils/           # 工具函数
├── tests/
│   └── test_smoke.py    # 冒烟测试
└── data/                # 样例数据
```

> `.env.example` 是关键:列出所有需要的 Key(值为占位),让评审/助教
>    一看就知道"哦,需要填这几个"。

### 知识点 3.2 各项目方向脚手架

教师在 GitHub 提供四份脚手架 repo(各方向一份),学员 fork 后开发:

```bash
# 示例:方向 A 脚手架
# README: 已能跑 "hello world" 问答
# 待做:接入真实文档、加工具、美化 UI
git clone https://github.com/xxx/capstone-A-template.git
```

> 脚手架的价值:启动不再是"空文件夹恐惧症",而是"已经能跑,接着改",
>    大幅降低启动阻力。

## 5. 第三讲(20 分钟) —— 组队 + 开发计划

### 知识点 5.1 组队规则(可选)

允许 1-2 人组队,超过 1 人需提交"分工说明":
- 每人至少承担一个独立模块(如 A 做后端,B 做前端)
- 代码提交需在 git log 里能看到双方贡献
- 答辩时双方都要能回答问题

> 组队 ≠ 抱大腿:评审会针对每人问自己负责的部分。

### 知识点 5.2 开发计划表

把 3 天(58/59/60)拆成具体任务:

```
Day 58(今天): 选题 + 搭骨架 + 跑通最小 demo
Day 59:       完成核心功能 1 + 写文档
Day 59 晚:    完成核心功能 2 + Docker 化
Day 60 上午:  准备演示 + 彩排 ≤ 5 分钟
Day 60 下午:  正式答辩
```

> 关键原则:**"最小 demo"今天必须能跑,不管多丑**。功能可以加,
>    但今天不能还在"配环境"。

## 易错点

1. **方向选太晚**: Day 58 上午必须定,否则起步晚 5 小时。
2. **计划写"完成功能"而非"跑通接口"**: 可验证的里程碑必须有
    HTTP / 命令行 / 截图作为证据。
3. **不 fork 脚手架**: 从零搭项目 2 小时就没了,脚手架省时间。
4. **忽略 `.env.example`**: 评审没有你的 Key 就跑不起来,必须给模板。
5. **3 天全写一行 commit**: Day 58-60 每天至少 3 个 commit,展示进度。

## 延伸题

- **(风险评估, ⭐⭐)**: 把开发计划里的每个里程碑标"乐观 / 可能 / 悲观"
    三份估算,悲观时间超过可用时间的里程碑就是风险。
- **(MVP 裁剪, ⭐⭐⭐)**: 列出功能的"必须有 / 最好有 / 可不做"三档,
    今天只做"必须有"的。
- **(导师制, ⭐⭐)**: 找一位 Day 59 已完成 demo 的同学做"技术顾问",
    卡住 30 分钟问他 —— 旁观者常一眼看出你的逻辑漏洞。
