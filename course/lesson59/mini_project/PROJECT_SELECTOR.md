# 期末作品选题指南

> **60 天 AI 就绪工程师课程 · 模块六（Day 58-60）**
>
> 本指南提供四个期末作品方向，每位学员需**任选其一**独立完成，并于 Day 60 进行答辩。

## 📋 选题流程

1. 仔细阅读四个方向的说明、技术栈推荐、验收 checklist 与参考案例
2. 结合自身兴趣、职业方向、技术储备评估匹配度

4. 选题确认后更换方向须提前 3 天向导师申请

## 🎯 选题原则

- **独立完成**：允许使用开源库和 AI 辅助工具，但核心逻辑必须自己实现
- **可运行交付**：作品必须是可运行的系统，不接受纯 PPT / 纯报告
- **体现所学**：尽量覆盖 Day 51-57 所学技术的组合应用

## 方向一：AI 应用方向 - RAG + Agent 智能助手

### 方向描述
构建一个基于大语言模型的知识库问答系统，能够对用户上传的文档进行解析、切片、向量存储，并在此基础上提供带检索增强生成（RAG）的问答；同时通过 Agent 集成外部工具，扩展问答能力边界。

### 技术栈推荐
- **框架**：LangChain / LlamaIndex
- **向量数据库**：ChromaDB / FISS / Qdrant
- **后端服务**：FastAPI + Uvicorn
- **前端 UI**：Streamlit / Slack Bot / FastAPI 原生页面
- **LLM 接入**：OpenAI / 智谱 / DeepSeek / Ollama（本地部署）
- **部署**：Docker + Docker Compose

### 验收 checklist

- [ ] 支持多种格式文档上传（PDF / Markdown / TXT），自动进行文本切片
- [ ] 文档内容经 Embedding 模型向量化后持久化存储到向量数据库
- [ ] 支持带上下文的 RAG 问答，能引用原文来源
- [ ] Agent 至少集成 **2 个工具**（如联网搜索、数据库查询、文件读写、API 调用）
- [ ] 提供 Web UI（Streamlit / Slack Bot / FastAPI 页面任选其一）
- [ ] 支持一键 Docker 化部署，提供 `docker-compose.yml`
- [ ] README 中包含架构图与清晰的运行说明

### 进阶加分项
- 支持多轮对话与历史记忆
- 支持多文档集合切换
- 引入 Reranker 重排序优化召回质量
- 实现评估模块（Recall / Faithfulness）

### 参考案例
- ChatPDF、Pinecone 官方 Demo、LlamaIndex RAG 教程
- 私人企业知识库、智能客服助手

## 方向二：LLM 微调方向 - 垂直领域客服模型

### 方向描述
针对某一细分领域（法律、医疗、金融、教育等），收集领域问答数据，通过 LoRA 等参数高效微调技术对开源基座模型进行微调，使其在该领域的回答质量显著提升，并通过量化指标对比微调前后效果。

### 技术栈推荐
- **微调框架**：HuggingFace Transformers + PEFT (LoRA / QLoRA)
- **数据集处理**：Datasets + Pandas
- **基座模型**：ChatGLM / Qwen / Llama（根据算力选择参数量级）
- **部署**：FastAPI 推理接口 + vLLM / TGII（可选加速）
- **评估**：BLEU / ROUGE / 人工评估对比表

### 验收 checklist

- [ ] 至少收集 **500 条**领域问答对，附完整的数据采集与清洗流程（含爬虫代码或数据源说明）
- [ ] 编写并执行 LoRA 微调训练脚本，保存 adapter 权重
- [ ] 提供微调前后的推理对比（至少 10 个典型 Query 的对比示例）
- [ ] 提交量化对比数据（BLEU、ROUGE 或人工打分表）
- [ ] 提供 FastAPI 推理接口，支持 `POST /chat` 调用微调后模型
- [ ] 撰写训练报告（参数量、loss 曲线图、训练时长、硬件环境、显存占用）
- [ ] README 中包含数据采集方法、微调流程与复现步骤

### 进阶加分项
- 实现 QLoRA 4-bit 量化微调，降低显存占用
- 使用 WandB / TensorBoard 可视化训练过程
- 提供 Gradio / Streamlit 在线体验 Demo
- 支持与 RAG 结合增强回答准确性

### 参考案例
- 法律问答小助手、医疗导诊助手、金融合规客服
- HuggingFace LoRA 官方教程、🤗 PEFT 文档

## 方向三：数据 + 爬虫方向 - 大规模数据采集与分析平台

### 方向描述
面向真实业务场景，从多个目标网站爬取大规模结构化数据，经过清洗、去重后持久化到关系型数据库，最后通过可视化 Dashboard 呈现分析结论。重点考察工程化数据采集与反爬应对能力。

### 技术栈推荐
- **爬虫框架**：Scrapy / Playwright / Requests + BeautifulSoup
- **异步任务**：Celery + Redis（定时调度、分布式爬取）
- **数据库**：PostgreSQL / MySQL / SQLite
- **数据处理**：Pandas + NumPy
- **可视化**：Streamlit / Plotly / ECharts / Matplotlib
- **部署**：Docker + Docker Compose

### 验收 checklist

- [ ] 至少爬取 **2 个站点**共 **10000 条**结构化数据
- [ ] 实现反爬处理：User-Agent 轮换、请求限速、重试策略（可拓展 IP 代理池）
- [ ] 编写完整的数据清洗与去重流程（缺失值处理、格式标准化、重复记录剔除）
- [ ] 数据持久化到关系型数据库，提供 ER 图或表结构说明
- [ ] 构建可视化分析 Dashboard（至少 5 类图表，覆盖分布、趋势、对比等维度）
- [ ] README 中包含爬虫架构图、数据字段说明与运行复现步骤

### 进阶加分项
- 实现定时增量爬取（Celery Beat 调度）
- 引入 Scrapy-Redis 分布式爬取
- 爬取数据接入 ML 模型进行预测
- 提供数据导出 API（CSV / JSON）

### 参考案例
- 电商平台商品价格监控、新闻舆情分析仪表盘
- 房产数据聚合分析、招聘市场趋势平台

## 方向四：ML 工程方向 - 端到端预测系统

### 方向描述
以经典机器学习解决真实预测问题（如销量预测、信用评分、用户流失预测等），使用 MLflow 管理实验过程，将最优模型服务化为 API，并通过 Prometheus 实现生产级监控。完整覆盖"数据 → 模型 → 服务 → 监控"全流程。

### 技术栈推荐
- **ML 框架**：scikit-learn / XGBoost / LightGBM
- **实验管理**：MLflow（追踪参数、指标、artifact）
- **后端服务**：FastAPI + Uvicorn
- **监控**：Prometheus + Grafana
- **数据集**：Kaggle 公开数据集或业务自采数据
- **部署**：Docker + Docker Compose

### 验收 checklist

- [ ] 提供完整的 EDA 报告（分布、相关性、异常值分析）与特征工程文档
- [ ] 至少对比 **3 个模型**，附性能对比表（准确率、召回率、F1 / RMSE 等业务相关指标）
- [ ] 使用 MLflow 进行实验追踪，记录每次训练的参数、指标与模型 artifact
- [ ] 提供 FastAPI 推理接口，支持 `POST /predict`，含输入校验（Pydantic）
- [ ] 实现监控指标采集：请求延迟（Latency）、吞吐量（QPS）、错误率
- [ ] 提供 Prometheus metrics 端点与可选的 Grafana Dashboard 截图
- [ ] README 中包含完整的项目架构图、复现步骤与模型选择理由

### 进阶加分项
- 实现 A/B 测试或模型版本切换机制
- 引入特征存储（Feature Store）概念
- 实现自动化训练流水线（定时重训）
- 提供模型解释性分析（SHAP / LIME）

### 参考案例
- 金融风控反欺诈预测、零售销量预测平台
- 用户流失预警系统、房价评估服务

## 🛠 脚手架说明

推荐所有方向的 starter template 结构如下（可根据方向微调）：

```
my_project/
├── README.md                # 项目首页文档（安装 / 运行 / 架构 / 亮点 / 难点）
├── requirements.txt         # 依赖清单（或 pyproject.toml）
├── .env.example             # 环境变量模板（不含真实密钥）
├── .gitignore               # 忽略 .env / __pycache__ / 数据等
├── data/                    # 数据目录（原始数据、清洗后数据）
├── notebooks/               # 探索性分析与实验 Notebook
├── src/                     # 核心业务代码
│   ├── __init__.py
│   ├── ...                  # 按模块拆分
├── tests/                   # 单元测试
├── Dockerfile               # 应用容器化
├── docker-compose.yml       # 多容器编排（含 DB / Redis / 监控等）
└── docs/                    # 架构图、API 文档、设计决策记录
```

### 脚手架使用建议

- **README.md** 是第一门面，务必写清楚"5 分钟启动"流程
- 敏感配置统一使用 `.env` + `python-dotenv` / Pydantic Settings 管理
- 代码按模块拆分，避免单个文件超过 300 行
- 至少编写 5 个核心函数的单元测试
