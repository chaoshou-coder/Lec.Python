# Day 60 · 期末答辩

> 本节时长: 6 小时(约 6 节 × 45 分钟)
> 前置: Day 59 已完成作品 + README + 5 分钟 demo 彩排;已准备录屏
>    备份;已按小组提交代码仓库地址 + .env.example
> 关键问题: 如何把 60 天的学习成果在 10 分钟内讲清、演明、答上?
    本节完成答辩、课程回顾与后续路线指引

---

## 0. 开场(10 分钟)

- **教师致辞**(3 分钟):回顾 60 天前 —— "你刚学 hello world",今天你
    已经开发并部署了 AI 应用。肯定每一位学员的努力。
- **答辩规则宣讲**(7 分钟):详细说明流程、评分标准、时间纪律(每
    人严格 10 分钟,超时扣分)、互评表怎么填。

---

## 1. 第一讲(15 分钟) —— 答辩流程详解

### 知识点 1.1 答辩时间分配(每人 10 分钟)

```
0:00-5:00   Demo 演示(评审打断可随时切换)
5:00-8:00   技术问答(针对作品提问)
8:00-9:00   互评环节(在互评表打分 + 一句反馈)
9:00-10:00  评委简评 + 换场
```

> 总人数 20 人 → 分 2 组并行,每组 1 位主讲教师 + 1 位助教评判,
>    全场约 2 小时完成。

### 知识点 1.2 答辩当天时间表

| 时间 | 内容 |
|---|---|
| 08:30-09:00 | 学员到场,最终调试,提交最终 commit |
| 09:00-09:10 | 开场 + 规则宣讲(本节) |
| 09:10-12:00 | A 组答辩(10 人) |
| 09:10-12:00 | B 组答辩(10 人)同步进行 |
| 12:00-13:30 | 午餐 + 合分 |
| 13:30-14:30 | 优秀作品展示(每组选 2 个) |
| 14:30-15:30 | 课程回顾 + 后续路线 |
| 15:30-16:00 | 颁奖 + 合影 + 结课 |

## 2. 答辩准备(15 分钟)

### 知识点 2.1 答辩前检查清单

```
□ 代码已 push,git 仓库可访问
□ README.md 包含快速开始 + 架构图 + 截图
□ .env.example 已填,密钥在本地 .env(不提交)
□ Docker 一键启动已验证(docker compose up -d)
□ /health 接口正常返回 200
□ 录屏备份(mp4)放在项目根目录
□ Demo 脚本已排练 ≤ 5 分钟
□ 回答 3 个预备问题:
  1. 你最大的技术难点是什么?怎么解决的?
  2. 如果给你一周时间,你下一步做什么?
  3. 项目里最不满意的点是什么?
```

### 知识点 2.2 预备的"必考题"

评审最常问的 5 个问题,学员必须提前准备:

1. **"为什么选这个技术方案,而不是另一个?"**(考察决策能力
    而非"我只会这一种")
2. **"遇到 bug 怎么定位的?"**(考察工程素养)
3. **"RAG 检索不出来怎么办?"** / **"Agent 无限循环怎么防?"**
    (考察对 Day 54-55 的掌握)
4. **"部署时遇到什么问题?"**(考察 Day 56-57 实践)
5. **"(对你作品某个细节)这个是怎么实现的?"**(区分自己做的 vs 抄的)

## 3. 评审标准(本节,10 分钟)

### 知识点 3.1 评分表(满分 100 分)

| 维度 | 权重 | 考察内容 |
|---|---|---|
| 功能完整性 | 30% | 声称的功能是否都能跑通 |
| 技术深度 | 25% | 是否合理选用工具 / 是否有原理讲解 |
| 演示表达 | 20% | 时间控制 / 逻辑清晰 / 有 demo 录像 |
| 文档质量 | 15% | README / 架构图 / 快速开始 |
| 互评参与 | 10% | 是否认真评价他人 + 反思 |

### 知识点 3.2 四个等级

| 等级 | 分数 | 标准 |
|---|---|---|
| 优秀 | 90-100 | 功能完整 + 运行流畅 + 口答清晰 + 文档完善 |
| 良好 | 75-89 | 功能基本完整,有少量缺陷但能讲清原因 |
| 合格 | 60-74 | 功能跑通,表达/文档略有欠缺 |
| 不合格 | <60 | 功能无法 demo / 明显抄袭 / 交互答不上来 |

> 不合格有 1 次补答辩机会(3 天后),学员不必过度紧张。

## 4. Day 60 上午:正式答辩(09:10-12:00,同步进行)

### 流程执行

各组严格按时间推进,设一位"计时员"举牌:
- 4 分钟 → "还剩 1 分钟"
- 5 分钟 → "请结束演示"
- 8 分钟 → "停止问答"
- 10 分钟 → "下一个"

### 评审记录

每组评审用标准化打分表记录:
- 3 个亮点(具体写到作品名 + 具体功能)
- 1 个改进点(可操作,不大而化之)
- 分数(4 项加总)

### 优秀候选

每组记录"可能优秀"的作品,下午集中展示前对比决定最终名单。

## 5. 优秀作品展示(13:30-14:30)

### 知识点 5.1 展示形式

每组选 2 个最高分作品(共 4 个),每人额外 **5 分钟精讲** —— 重点讲
"最有技术含量的部分",让全体学员看到同龄人能做到的水平。

### 知识点 5.2 颁奖

| 奖项 | 数量 | 奖励(建议) |
|---|---|---|
| 最佳技术深度奖 | 1 | ￥300 课程代金券 |
| 最佳 UI/UX 奖 | 1 | ￥200 京东卡 |
| 最佳工程实践奖 | 1 | 《流畅的Python》纸质书 |
| 最具创意奖 | 1 | ￥150 外卖券 |
| 最佳表达奖 | 1 | 讲师 1v1 简历指导 30 分钟 |

> 奖金/奖品不是目的 —— 仪式感对成人学员同样重要,强化"我能做到"。

## 8. 结课(15 分钟)

### 流程

- **学员感言**(5 分钟):自愿报名,1-2 人,每人 1-2 分钟讲最大收获
    或最崩溃的瞬间(笑点 + 泪点都行)
- **教师寄语**(3 分钟):肯定每位学员的付出,强调"这只是起点"
- **合影**(3 分钟):拉开课程结课横幅,全体合影
- **结课证书发放**(4 分钟):每人一份 PDF 电子证书(含姓名、课程名、
    日期、作品链接)

## 易错点(答辩当天提醒)

1. **迟到**: 提前 30 分钟到场调试,不要答辩前 1 分钟才 clone。
2. **API Key 没充够**: 答辩前在控制台查余额,至少留 5 次 demo 量。
3. **没写 README 快速开始**: 评审克隆后 3 分钟跑不起来就直接 pass。
4. **demo 超时被切**: 练习时严格 5 分钟,宁可少讲不要超时。
5. **被问就慌**: 不知道就坦诚说"这里我还没深入了解,计划后续学",
    比胡编更有工程师气质。
