# 期末作品答辩评分标准

> **60 天 AI 就绪工程师课程 · 模块六（Day 58-60）**
>
> Day 58-60 为期末作品 + 答辩，每位学员进行 **10 分钟现场展示 + 5 分钟评委问答**。
> 本评分标准提前公开，学员可针对性准备。

## 📋 答辩流程

| 环节 | 时长 | 说明 |
| --- | --- | --- |
| 点名签到 | 2 分钟 | 确认答辩顺序 |
| 作品展示 | 10 分钟 | 项目背景 → 架构 → 核心演示 → 亮点与难点 |
| 评委问答 | 5 分钟 | 技术细节、设计决策、扩展思考 |
| 评委打分 | 3 分钟 | 独立打分、现场不公布 |
| 中场休息 | 2 分钟 | 下一位学员准备 |

### 展示建议

- 前 2 分钟讲清 **Why**（解决什么问题）和 **What**（做了什么）
- 中间 5 分钟 **现场演示**核心功能，展示真实运行效果
- 后 3 分钟讲 **How**（技术选型理由、架构设计、难点与解决方案）
- 务必**提前演练**，控制时间，超时将被打断

---

## 评分维度总览

| 维度 | 权重 | 核心考察点 |
| --- | --- | --- |
| 功能完整性 | 30% | 需求覆盖度、健壮性、边界处理 |
| 技术深度 | 30% | 技术理解、架构设计、工程规范 |
| 演示表达 | 20% | 逻辑清晰、重点突出、节奏把控 |
| 文档质量 | 20% | README 规范、可复现性、架构呈现 |
| **合计** | **100%** | — |

附加：**加分项最多 +10 分**；扣分项从总分中扣除（详见后文）。

---

## 1. 功能完整性（30%）

> 考察项目是否完整实现了既定需求，核心功能是否可正常运行，对边界情况与异常输入的处理是否合理。

### 评分细则

| 等级 | 分数 | 标准描述 |
| --- | --- | --- |
| **优秀** | 30 分 | 全部功能实现；边界处理完善；演示全程未出现崩溃或明显 bug |
| **良好** | 21 分 | 核心功能完整；次要功能有小瑕疵但不影响整体演示 |
| **中等** | 15 分 | 基本可运行，但存在明显的功能缺失或演示需跳过部分模块 |
| **及格** | 9 分 | 功能残缺严重，仅能演示少量流程，演示过程出现多次错误 |
| **不及格** | 0 分 | 无法运行或演示完全失败 |

### 评分关注点

- [ ] 核心功能是否完整实现（对照验收 checklist）
- [ ] 异常输入是否有合理处理（空输入、超长输入、非法字符等）
- [ ] 演示过程中是否出现崩溃或逻辑错误
- [ ] 功能是否超出了"demo 级别"，有一定真实可用性
- [ ] UI 交互是否流畅（输入响应、错误提示、加载状态）

### 常见扣分情形

- 演示现场发现功能跑不通
- 未做输入校验导致异常崩溃
- 核心功能依赖演示者手动跳过错误步骤
- 明显功能缺失且未说明原因

---

## 2. 技术深度（30%）

> 考察学员对 Day 51-57 所学技术栈的理解深度，以及能否将多项技术进行合理组合应用，是否有清晰的工程化思维。

### 评分细则

| 等级 | 分数 | 标准描述 |
| --- | --- | --- |
| **优秀** | 30 分 | 体现 Day 51-57 多项技术的深入理解；系统架构图清晰；技术选型有明确理由 |
| **良好** | 21 分 | 正确使用主要技术栈；能解释关键技术点的作用 |
| **中等** | 15 分 | 浅层使用技术；缺乏对原理的认识；仅停留在"能跑"层面 |
| **及格** | 9 分 | 多为 copy 范例代码，对技术细节解释不清 |
| **不及格** | 0 分 | 无技术含量，纯套壳或完全依赖 AI 生成且无法解释 |

### 评分关注点

- [ ] 是否提供了清晰的系统架构图（组件、数据流、依赖关系）
- [ ] 是否能讲清核心技术选型理由（为什么选 Chroma 而不是 FAISS？为什么选 LoRA？）
- [ ] 是否体现了 Day 51-57 所学技术的组合（RAG + Agent、MLflow + Prometheus 等）
- [ ] 访谈环节能否准确回答技术细节问题
- [ ] 是否体现了工程化思维（模块拆分、配置管理、错误处理、日志）

### 常见扣分情形

- 无法解释技术选型的理由
- 架构图缺失或过于粗糙
- 对关键技术（如 Embedding、CoT、Rerank）无法说清原理
- 访谈环节暴露对代码不熟悉

---

## 3. 演示表达（20%）

> 考察学员的现场表达能力，是否能逻辑清晰、重点突出地呈现项目，时间控制是否得当。

### 评分细则

| 等级 | 分数 | 标准描述 |
| --- | --- | --- |
| **优秀** | 20 分 | 逻辑清晰；讲清 Why / What / How；时间控制精准；有互动有节奏 |
| **良好** | 14 分 | 能讲清项目内容与核心功能；逻辑基本通顺 |
| **中等** | 9 分 | 照念笔记或念 PPT 文字；缺乏重点；前后逻辑跳跃 |
| **及格** | 4 分 | 无法讲清做了什么；表达混乱或严重超时 |
| **不及格** | 0 分 | 放弃演示或完全无法表达 |

### 评分关注点

- [ ] 是否有清晰的项目背景引入（Why）
- [ ] 是否现场演示真实功能（而非仅展示截图）
- [ ] 是否突出项目的亮点与难点
- [ ] 是否在规定时间内完成（10 分钟 ± 1 分钟）
- [ ] 语速、肢体语言、眼神交流是否自然

### 加分表达方式

- 主动暴露一个技术难点并说明解决方案
- 演示时展示"真实版本"而非"演示特供版本"
- 使用对比（before / after）展示效果提升

---

## 4. 文档质量（20%）

> 考察项目 README 与配套文档的完整性、规范性与可复现性，这是工程师基本功的直接体现。

### 评分细则

| 等级 | 分数 | 标准描述 |
| --- | --- | --- |
| **优秀** | 20 分 | README 结构完整（安装 / 运行 / 架构 / 亮点 / 难点）；含架构图；5 分钟可复现 |
| **良好** | 14 分 | 有基本说明，包含核心运行步骤 |
| **中等** | 9 分 | 只有标题或一句话简介；缺少关键运行说明 |
| **及格** | 4 分 | 文档存在但缺失关键信息（如缺依赖安装、缺 API Key 配置说明） |
| **不及格** | 0 分 | 无任何文档或仅有 Auto-Generated README |

### 评分关注点

- [ ] README 是否包含完整项目标题、简介与功能描述
- [ ] 是否提供 Quick Start（5 分钟内可跑通）
- [ ] 是否包含清晰架构图（推荐使用 Mermaid / draw.io）
- [ ] 是否列出依赖安装步骤（`pip install -r requirements.txt` 等）
- [ ] 是否说明环境变量 / API Key 配置方式
- [ ] 是否展示项目亮点与难点总结
- [ ] 是否包含 API 文档或功能截图

### README 推荐结构

```markdown
# 项目名称

## 1. 项目简介
## 2. 功能特性
## 3. 技术栈
## 4. 系统架构
## 5. Quick Start
   ## 5.1 环境要求
   ## 5.2 安装依赖
   ## 5.3 配置环境变量
   ## 5.4 启动服务
## 6. 功能截图 / Demo
## 7. 亮点与难点
## 8. 后续规划
## 9. 致谢 / 参考
```

## ✨ 加分项（最多 +10 分）

具备以下亮点可获额外加分，但加分总分不超过 10 分。

| 加分项 | 分值 | 认定标准 |
| --- | --- | --- |
| 上线真实云服务 | +5 分 | 部署到阿里云 / AWS / Vercel 等，提供真实可访问的 URL |
| 引入 MCP 或 A2A 等协议 | +3 分 | 实现 Agent 间通信或工具调用的标准协议集成 |
| 有完整的 CI/CD | +2 分 | 配置 GitHub Actions / GitLab CI，自动化测试与部署 |
| 开源社区贡献 | +2 分 | 项目获得 Star / Fork 或提交 PR 到上游项目 |
| 创新性功能 | +2 分 | 在基础需求之外实现了有亮点的创新功能 |

> 注：加分项需在答辩时提供可验证证据（URL、截图、CI 运行记录等）。

## ❌ 常见扣分项

以下情形将从总分中直接扣除相应分数。

| 扣分项 | 分值 | 说明 |
| --- | --- | --- |
| API Key 硬编码提交到 git | -10 分 | 属于安全红线，一律扣 10 分 |
| 未使用 .env 管理敏感配置 | -5 分 | 存在密钥泄露风险 |
| README 缺运行说明 | -5 分 | 项目可复现性为零 |
| 无法复现运行 | -10 分 | 评委 10 分钟内无法跑通项目 |
| 代码全量 AI 生成且无法解释 | -10 分 | 技术深度维度同时记 0 分 |
| 抄袭他人作品 | -30 分 | 取消答辩资格，报教务处理 |

> 注：扣分项可叠加，若扣至 0 分以下按 0 分计。

## 📊 总分换算

最终总分 = 功能完整性 + 技术深度 + 演示表达 + 文档质量 + 加分项 - 扣分项

| 等级 | 分数区间 | 含义 |
| --- | --- | --- |
| **A** | ≥ 90 分 | 优秀：推荐为课程标杆作品 |
| **B** | 80 - 89 分 | 良好：达到预期目标 |
| **C** | 70 - 79 分 | 中等：基本满足要求，有提升空间 |
| **D** | 60 - 69 分 | 及格：勉强通过，需补强短板 |
| **F** | < 60 分 | 不及格：需重修或补交 |

### 等级与课程认证

- **A 等**：获得课程优秀证书 + 推荐至合作企业实习
- **B 等**：获得课程结业证书
- **C 等**：获得课程结业证书，建议复盘总结
- **D 等**：获得课程结业证书（标注补考），需提交复盘报告
- **F 等**：不颁发结业证书，建议重修

## 🎯 答辩准备清单

学员在答辩前请逐项确认：

- [ ] 项目在本地可一键运行（`docker compose up` 或 `python main.py`）
- [ ] 演示流程已完整演练 ≥ 2 次，时间在 9-11 分钟
- [ ] 准备好离线演示方案（以防现场网络问题）
- [ ] 笔记本电脑充满电，携带电源适配器和转接头
- [ ] PPT 导出为 PDF 备份（以防播放设备兼容问题）
- [ ] 提前了解评委可能提问的 3 个技术问题并准备答案
- [ ] API Key 已从代码中移除，仅通过 `.env` 传入

---

> 💡 **温馨提示**：答辩不是"汇报你做了什么"，而是"说服评委你解决了什么问题"。
> 从 Why 出发，让评委理解你的思考过程，比炫技术更重要。
